# rawmem 0.6.2

rawmem 0.6.2 closes the append-only boundary for ledgers whose historical hash
chain contains known link breaks. It archives the original bytes instead of
rewriting hashes and starts a clean, linked active ledger at the same path.

## Sealed archives

- `rawmem seal --yes` / `seal_ledger()` performs a full verification under the
  active writer lock.
- Only `previous_hash_mismatch` is an accepted historical breakpoint. JSON,
  schema, duplicate-id, content-hash, and concurrent-change failures block the
  seal before the source moves.
- The archive bytes are SHA-256 checked before and after the move and marked
  ReadOnly on Windows.
- Each archive receives a versioned manifest, full verification report, and
  hash-pinned breakpoint list.
- The original active path starts with a valid `ledger_transition` event and
  remains the only default capture target.
- The metadata-only archive registry is derived and atomically replaced; each
  archive manifest is authoritative.

## Read boundaries

- `verify_ledger()` is now strictly read-only and never creates or refreshes a
  lock or state sidecar.
- Active readers reject sealed archive paths.
- `iter_archive_events()` and CLI `--archive` are explicit archive-only reads.
- Every archive query validates the complete archive SHA before returning even
  a limited batch.
- Archive reads continue over manifest-pinned historical breakpoints and emit
  structured integrity warnings.
- Archive CLI/API reads default to metadata-only projection; summary and full
  event bodies require explicit selection.

## Compatibility

The event schema remains `rawmem.event.v1`. Existing active ledgers and cursors
remain valid. `rotate_ledger()` / `rawmem rotate --yes` now delegate to the
safer sealed-archive transaction and remain available as compatibility aliases.
